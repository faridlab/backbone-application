# Maintainer Guide

> **Reader:** maintainer — you extend the service without breaking its conventions. **Mode:** how-to.
> Assumes you've read [architecture/layers.md](./architecture/layers.md) and
> [architecture/ddd.md](./architecture/ddd.md). This page is the *procedure*, not the theory.

The golden rule you'll apply constantly: **the schema YAML upstream is the source of truth; this
service composes.** Most "add a feature" tasks are a module change plus a version bump — not a code
change here. This guide covers the cases where you *do* touch this repo.

## The regeneration model (know this cold)

`metaphor schema generate [--force]` regenerates the bulk of `src/`, `migrations/`, `config/`, and
`tests/` from `schema/models/*.model.yaml`. Three — and only three — mechanisms preserve
hand-written code. Know which protects what *before* you write anything inside a generator tree.

| # | Mechanism | Protects | Where |
|---|---|---|---|
| 1 | `metaphor.codegen.yaml` → `user_owned:` | whole files inside a generator tree | [`metaphor.codegen.yaml`](../metaphor.codegen.yaml) |
| 2 | `// <<< CUSTOM … // END CUSTOM` markers | regions *inside* a generator-emitted file | e.g. `src/lib.rs`, aggregator `mod.rs` |
| 3 | `-- Generated by metaphor-schema` header | migration sweep (files *with* the header are deletable) | `migrations/` |

Full detail lives in [`CLAUDE.md` → Regen safety](../CLAUDE.md). The failure mode is silent: a
hand-written file inside a generator tree with **no** `user_owned:` entry and **no** CUSTOM markers
is *deleted* on the next `--force`. When in doubt, add the `user_owned:` entry.

### After every `metaphor schema generate --force`

1. `git status` — review every `D` (deleted) and `M` (modified). Anything unexpected → stop and flag.
2. `cargo check` — verify it compiles **before** running anything else.
3. Confirm the paths you listed in `metaphor.codegen.yaml` do **not** appear in `git status`. If one
   does, your glob doesn't match.

## Add-a-feature: the decision tree

```
Is it business logic (an entity, invariant, rule, endpoint for a domain concept)?
├── YES → it does NOT go here.
│         Edit schema/models/*.model.yaml in the OWNING module upstream,
│         regenerate there, bump ref: in the workspace metaphor.yaml,
│         metaphor sync --update, metaphor migration run-all.
│
└── NO → is it composition / wiring / a cross-context orchestrator?
         └── YES → it goes here. Continue below.
```

### Case A: wire a new module into the binary

The most common local change. See the [developer guide recipe](./developer-guide.md#recipe-add-a-domain-module-to-the-service)
for the mechanics. Files you touch: [`Cargo.toml`](../Cargo.toml) (dep at the shared tag) and
[`src/main.rs`](../src/main.rs) (`Module::builder().with_database(pool).build()?` + `.merge(router)`).
Both are `user_owned:` — safe from regen.

### Case B: add a custom application service (no schema model)

A service with no entity behind it (e.g. a cross-context orchestrator) legitimately lives next to
generated code in `src/application/services/` (the **hand-written**, plural folder — not
`src/application/service/`, which is generated). Two ways to keep it:

- Put it in the hand-written folder (plural / snake_case), **or**
- If it must sit inside a generator-owned folder, add its exact path to `user_owned:` in
  [`metaphor.codegen.yaml`](../metaphor.codegen.yaml).

The generated-vs-hand-written naming convention (bare/singular = generated, plural/snake_case =
hand-written; definitive check is the `mod.rs` header) is in [architecture/layers.md](./architecture/layers.md).

### Case C: edit inside a generator-emitted aggregator file

Some files (`src/lib.rs`, `src/application/service/mod.rs`, `src/domain/repositories/mod.rs`) are
rewritten on regen but preserve content between markers:

```rust
// <<< CUSTOM
pub mod my_orchestrator;
pub use my_orchestrator::*;
// END CUSTOM
```

The block **must contain complete syntactic units** — full `pub mod foo;` / `pub use foo::*;`
statements, never bare identifiers or field fragments. Edits outside the markers are wiped.

### Case D: add a hand-written migration

Prefer regenerating from the module's schema YAML. When you genuinely need a manual migration:

- Put it in `migrations/manual/**` (listed under `user_owned:`), **or**
- Ensure the file has **no** `-- Generated by metaphor-schema` header (headerless files survive the sweep).
- **Gotcha:** a file that started as generator output still carries the marker and *will* be swept.
  Remove the marker line, move it to `migrations/manual/`, or pin its path under `user_owned:`.

## Where new code goes (per layer)

The full map is [architecture/layers.md](./architecture/layers.md). The maintainer's short version:

| You're adding… | It goes in… | Regen note |
|---|---|---|
| Module wiring | [`src/main.rs`](../src/main.rs) | `user_owned:` |
| A tower middleware | [`src/middleware/`](../src/middleware/) | `user_owned:` |
| A cross-context orchestrator | `src/application/services/` (hand-written) | plural folder or `user_owned:` |
| A custom infra adapter | `src/infrastructure/…` | add path to `user_owned:` |
| A manual migration | `migrations/manual/` | `user_owned:` |
| Anything schema-derived | **upstream module's schema YAML** | not here |

## Versioning & release

- **Bump a module:** update its dep tag in [`Cargo.toml`](../Cargo.toml), `metaphor dev db migrate`,
  `metaphor dev test`. All `backbone-*` crates track a single tag (currently `v2.1.0`) — keep them aligned.
- **Cut a release:** `git tag v0.5.2 && git push --tags`. CI
  ([`.github/workflows/release-image.yml`](../.github/workflows/release-image.yml)) builds and pushes
  the image to GHCR on any `v*` tag. Deploy with `./scripts/deploy-service.sh backbone-app v0.5.2 SERVICE_TAG`.
- **Validate prod env first:** `./scripts/preflight-prod.sh` checks both env files
  (`.env.prod` at the root for runtime secrets, `deployment/.env.prod` for orchestration) before any
  sync to the VPS. Operator runbook: [`deployment/README.md`](../deployment/README.md).

## Guardrails (the MUST/NEVER that catch maintainers)

- **NEVER** put business logic in [`src/main.rs`](../src/main.rs). It belongs in a module.
- **NEVER** hand-roll `axum::Router::new().route(...)` for CRUD — use `BackboneCrudHandler`.
- **NEVER** run `cargo build`/`cargo test` from the *workspace* root — each project has its own
  `Cargo.toml`; use `metaphor build`/`metaphor test`. (Inside this standalone repo, `cargo` is fine.)
- **NEVER** edit anything under a workspace's `modules/*` — read-only clones, wiped by `metaphor sync`.
- **MUST** split any file over 500 lines along a real seam — never `foo_part2.rs`.
- **MUST** verify before "done": `metaphor lint check && metaphor dev test`, then read the diff for
  cross-layer imports and unexpected deletions.

Full behavioral rules: [architecture/ai-guidelines.md](./architecture/ai-guidelines.md).
